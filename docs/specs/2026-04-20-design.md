# robot-md-pendant — v0.1 Design

> **Status:** Draft, pending author review. Created 2026-04-20.
>
> **Scope:** Initial public release (v0.1). Touchscreen + voice pendant that bridges a Waveshare ESP32-C6 Touch AMOLED 1.8" to a Raspberry Pi 5 running the Claude Agent SDK and `robot-md-mcp`. Targets the SO-ARM101 via the existing `robot-md` stack.
>
> **Hardware note:** an earlier draft of this spec targeted ESP32-S3. The pendant on hand is ESP32-C6 (RISC-V single-core, Wi-Fi 6 + BLE5, no PSRAM on most variants, one I2S peripheral). ESP-SR on-device speech is not available on C6 — but that's consistent with our design choice to run wake-word on the Pi. See the "Firmware" section below.
>
> **Predecessor:** none — new project.
>
> **Sibling projects:** [`robot-md`](https://github.com/RobotRegistryFoundation/robot-md) (ROBOT.md spec + Python CLI), [`robot-md-mcp`](https://github.com/RobotRegistryFoundation/robot-md-mcp) (MCP server exposing ROBOT.md to any MCP client). The pendant uses `robot-md-mcp` as its primary interface to the robot.

---

## Summary

`robot-md-pendant` turns a $35 ESP32-S3 AMOLED touch-display into a voice- and tap-driven remote for any `ROBOT.md`-declared robot. A small Python service (`pendantd`) runs on the same Raspberry Pi that owns the robot's USB bus; it hosts a WebSocket for the pendant, keeps a Claude Agent SDK session alive, and registers `robot-md-mcp` as a tool server. The pendant is a thin frontend: it renders state, ships user intent (button presses, voice, typed prompts), and shows Claude's tool-call stream live. No robot logic, no secrets, no Claude awareness lives on the pendant.

**One-line pitch:** "Talk to your robot from across the room, with a status dashboard in your hand."

**The user-visible wins:**
- Zero-friction control: tap **Home**, tap **Pick**, tap **Release** — three buttons, no CLI.
- Conversational control: hold the mic, say "pick the red LEGO and drop it in the bowl," Claude plans, robot acts.
- Glanceable status: live thumbnail from the OAK-D, current pose, last detection, robot health — all on the 1.8" screen.
- Safe-by-default: heartbeat watchdog halts the arm if the pendant vanishes. Soft-stop button on the UI. (Hardware e-stop is a v2 concern; see "Known limits.")

## Architecture

```
 ┌───────────────────────────┐                 ┌──────────────────────────────────┐
 │ ESP32-C6 Touch AMOLED 1.8 │                 │ Raspberry Pi 5 (robot host)      │
 │                           │                 │                                  │
 │  LVGL UI:                 │  WebSocket      │  pendantd (Python, systemd)      │
 │   ├─ status strip         │  JSON + binary  │   ├─ WebSocket server (:8765)    │
 │   ├─ thumbnail (OAK-D)    │◄───────────────►│   ├─ heartbeat watchdog          │
 │   ├─ preset buttons       │  ws://pi:8765   │   ├─ Claude Agent SDK session    │
 │   ├─ chat pane            │                 │   ├─ button/voice config loader  │
 │   └─ soft-stop button     │                 │   ├─ Whisper STT (whisper.cpp)   │
 │                           │                 │   ├─ piper-tts                   │
 │  Mic (I2S, onboard)       │                 │   └─ OpenWakeWord (optional)     │
 │  Speaker (I2S, onboard)   │                 │                │                 │
 │  VAD gate                 │                 │                ▼                 │
 │  10Hz heartbeat           │                 │  MCP subprocess: robot-md-mcp    │
 │                           │                 │    ├─ execute_capability         │
 │  Wi-Fi STA (home SSID)    │                 │    ├─ execute_task               │
 │   ↓ fallback              │                 │    ├─ estop / estop_clear        │
 │  Wi-Fi AP (self-hosted)   │                 │    └─ validate / render          │
 └───────────────────────────┘                 │                │                 │
                                               │                ▼                 │
                                               │  /dev/ttyACM0 (SO-ARM101)        │
                                               │  OAK-D  (thumbnail source)       │
                                               │  ROBOT.md (authority)            │
                                               └──────────────────────────────────┘
```

**Design principles:**

- **The pendant is dumb.** Rendering + input only. Survives Claude model upgrades, MCP tool changes, and `robot-md` releases with no firmware change.
- **ROBOT.md is the authority.** Capabilities, safety gates, and preset pose definitions live in ROBOT.md. The pendant never hard-codes any of them.
- **Zero-friction first.** Sensible defaults for everything (button set, voice opt-in, network fallback). Precision-adding knobs (custom wake word, tuned TTS voice, hardware e-stop wiring) are opt-in.
- **Safety-first on state transitions.** Any unexpected state → Pi halts the arm via MCP `estop`, then diagnoses. No "are you sure?" dialogs.

## Components

Four pieces, each with a single job.

### 1. `firmware/` — ESP32-C6 firmware

ESP-IDF + LVGL. Builds for the Waveshare ESP32-C6-Touch-AMOLED-1.8 board (368×448 AMOLED, capacitive touch, onboard I2S mic + speaker). RISC-V single core @ 160MHz, Wi-Fi 6, BLE5, no PSRAM on standard variants. ESP-SR is not available on C6; wake-word runs Pi-side.

**Responsibilities:**
- LVGL UI with four zones: status strip (top), thumbnail/dashboard (middle), chat pane (middle, swappable), button grid + soft-stop (bottom).
- Wi-Fi manager: tries saved STA credentials → on 15s timeout, starts AP `robot-md-pendant-<mac6>` with captive portal for SSID + Pi IP entry.
- WebSocket client: reconnect w/ backoff, JSON line protocol + binary audio frames.
- 10Hz heartbeat (WS frame).
- I2S mic capture: 16kHz mono PCM, 20ms chunks.
- I2S speaker output: buffered PCM playback with a small jitter queue.
- VAD gate: cheap RMS + zero-crossing heuristic, opens when speaking, closes on silence. Gates audio upload — audio leaves the pendant only while voice is detected.

**Explicitly NOT in scope for v0.1:**
- On-device wake word (handled Pi-side via OpenWakeWord).
- On-device STT / TTS (handled Pi-side).
- Robot-specific logic of any kind.

### 2. `pendantd/` — Raspberry Pi Python service

Python 3.11+, packaged as a systemd unit. Single process, async (`asyncio` + `websockets`).

**Responsibilities:**

- **WebSocket server** on `:8765` (configurable). One session per pendant, keyed by `pendant_id = sha256(mac)[:8]`.

- **Heartbeat watchdog.** 3 missed heartbeats (>300ms) → call `robot-md-mcp::estop` tool + emit `pendant_lost` event. On reconnect: call `estop_clear` and push fresh `hello` snapshot.

- **Claude Agent SDK session** per pendant, with `robot-md-mcp` registered as an MCP server (stdio transport). System prompt modeled on the robot-md workspace CLAUDE.md convention — tells Claude how to use capabilities, which safety gates to respect, when to dry-run first.

- **Button config loader.** Watches `~/.config/robot-md-pendant/buttons.yaml`. On change, pushes updated button set to connected pendants.

- **Voice pipeline:**
  - Incoming `audio_in` frames → buffer.
  - If wake-word mode enabled: OpenWakeWord watches buffer for "Claude" (TFLite model in `~/.config/robot-md-pendant/wake-models/`).
  - If PTT mode: whole PTT window goes to Whisper on button release.
  - Whisper (whisper.cpp, `ggml-base.en.bin`) transcribes.
  - Transcript feeds the Agent SDK turn as if the user had typed it.
  - Agent SDK reply text → piper-tts (voice `en_US-lessac-medium` by default) → PCM → `audio_out` frames.
  - Barge-in: if VAD fires while speaker is playing, cancel the piper stream and emit `voice_state{speaking:false}`.

- **Thumbnail producer.** Every 500ms while the dashboard tab is visible on the pendant, grab a frame from the OAK-D via `robot-md`'s vision layer, downscale to 160×120, JPEG-encode, push as `thumbnail{b64}`. Paused when the pendant reports `dashboard_visible:false`.

- **Tool-call streaming.** The Agent SDK emits tool-call events (start, args, result). These are mapped 1:1 to pendant `tool_call{name, args, status, summary}` events so the user sees what's happening live.

### 3. `config/` — user-editable on the Pi (ships defaults)

**`buttons.yaml`** — the preset button grid. Two button types:

```yaml
buttons:
  # Type 1: "prompt" — Agent SDK plans the action
  - label: "Home"
    icon: "home"
    type: "prompt"
    prompt: "Home the arm"

  # Type 2: "direct_mcp" — bypass the LLM, call the tool directly (lower latency)
  - label: "Pick LEGO"
    icon: "grab"
    type: "direct_mcp"
    tool: "execute_capability"
    args: {capability: "pick", target: "red_lego"}

  - label: "Release"
    icon: "release"
    type: "direct_mcp"
    tool: "execute_capability"
    args: {capability: "release"}

# Soft-Stop is always present; not configurable; always type: direct_mcp → estop.
```

**`voice.yaml`** — voice pipeline config:

```yaml
wake_word: false              # opt-in per zero-friction posture
wake_model: "claude.tflite"   # path relative to wake-models/; default stub shipped
sticky_ptt: false             # tap-to-toggle vs press-and-hold
tts_voice: "en_US-lessac-medium"
barge_in: true
```

**`pendantd.yaml`** — service config:

```yaml
listen: "0.0.0.0:8765"
anthropic_api_key_env: "ANTHROPIC_API_KEY"  # reads from env, never in config file
robot_md_mcp_cmd: "npx robot-md-mcp --robot /path/to/ROBOT.md"
thumbnail_fps: 2
```

### 4. `docs/protocol.md` — WebSocket protocol, versioned

Single source of truth for message shapes, consumed by both firmware and `pendantd`. JSON schemas live under `schemas/v1/` and are validated in CI against a fixture corpus.

## Data flow

**Protocol shape:** single WebSocket per pendant. JSON text frames for control; binary frames for audio, prefixed with a 1-byte type tag (`0x01`=audio_in, `0x02`=audio_out). Every JSON frame carries `{"v":1, "type":..., ...}`.

**Flows:**

| # | Flow | Direction | Trigger | Key messages |
|---|------|-----------|---------|--------------|
| 1 | Connect | both | pendant boot → WS opens | Pi → `hello{buttons, voice_state, wifi_mode, robot_status}` |
| 2 | Heartbeat | pendant→Pi | every 100ms | `heartbeat{seq}`; Pi watchdog resets |
| 3 | Pendant lost | Pi→robot | 3 missed heartbeats | Pi invokes MCP `estop`; on reconnect: `estop_clear` + fresh `hello` |
| 4 | Button press (prompt) | both | user taps a `type: prompt` button | `button_press{id}` → Pi feeds `prompt` into Agent SDK → tool-call stream → `status` |
| 5 | Button press (direct) | both | user taps a `type: direct_mcp` button | `button_press{id}` → Pi invokes MCP tool directly → `tool_call{...}` → `status` |
| 6 | Chat (text) | both | user types + sends | `chat_prompt{text}` → Agent SDK turn → per-tool `tool_call{...}` live → turn-end `chat_message{role:assistant, text}` |
| 7 | Voice in (PTT) | pendant→Pi | mic button held | binary `audio_in` frames → on release, Whisper finalizes → feeds flow 6 |
| 8 | Voice in (wake) | pendant→Pi | VAD fires | `audio_in` frames → OpenWakeWord detects "Claude" → endpointed → Whisper → flow 6 |
| 9 | Voice out | Pi→pendant | Agent SDK turn ends with text reply | piper-tts PCM → binary `audio_out` frames |
| 10 | Thumbnail | Pi→pendant | every 500ms while dashboard visible | `thumbnail{b64_jpeg, w, h}` |
| 11 | Config reload | Pi→pendant | filewatcher on `buttons.yaml` / `voice.yaml` | `buttons{...}` or `voice_state{...}` |
| 12 | Soft-stop | both | user taps big red button | `soft_stop` → MCP `estop` + cancel in-flight Agent SDK turn → `status{estopped:true}` |
| 13 | Barge-in | both | VAD fires while speaker playing | pendant halts playback + sends `barge_in` → Pi cancels piper stream |

**Reconnect contract:** pendant backoff 1s/2s/5s/10s (capped). On reconnect, Pi pushes a fresh `hello` snapshot — no message replay. Agent SDK session persists across reconnects (keyed by `pendant_id`, not WS connection).

## Error handling

**Invariant:** robot motion only proceeds while heartbeat + WS are both healthy *and* the last tool call returned cleanly. Any deviation → MCP `estop`, then diagnose.

| Failure | Detection | Pi response | Pendant UX |
|---|---|---|---|
| Pendant STA can't connect | 15s timeout | — | Auto-switch to AP + captive portal |
| WS drops mid-session | WS close event | Halt Agent SDK turn; start watchdog | Backoff reconnect, banner "Pi unreachable" |
| 3 missed heartbeats | Pi watchdog (>300ms) | MCP `estop` + `pendant_lost` log | On reconnect: `estop_clear` + fresh `hello` |
| MCP tool call returns error | tool result `isError:true` | Emit `tool_call{status:error, message}` | Toast + expandable error detail |
| Servo latched (STS3215 overload) | `robot-md doctor` poll shows joint missing | `robot_status=servo_latched{joint}`; refuse further motion | Red banner: "Wrist flex latched — physical power cycle required" |
| USB bus disconnect (`/dev/ttyACM0` gone) | udev event / MCP call fails | `robot_status=bus_disconnected` | Dashboard red, buttons disabled, chat still works |
| Anthropic API 429/529 | Agent SDK retry exhausted | `claude_unavailable{retry_in}` | "Claude is busy, retrying in Ns" — direct_mcp buttons still work |
| Anthropic API auth error | 401 | `claude_config_error` (no retry) | Modal: "API key invalid — check pendantd config" |
| Pi offline (no internet) | DNS/network fail | Same as 429 path | "Offline — preset buttons still functional" |
| Whisper fails / times out | exit or >10s | `voice_error{stage:stt}` | Toast "Didn't catch that — try again" |
| OpenWakeWord crashes | process death | Auto-restart; 3 crashes/min → disable wake | Wake indicator greys out; PTT still works |
| piper-tts fails | exit | Skip voice-out; text still in chat | No speech; text reply only |
| `buttons.yaml` invalid | YAML parse fails on filewatch | Keep last-valid config, log error | Banner "Config error — using previous buttons" |
| ESP32 brown-out / crash | reboot | Heartbeat timeout path | Reconnect after boot (~2-3s) |

**What's deliberately NOT handled:** Pi itself crashing. The watchdog for that is systemd auto-restart; pendant shows disconnect, reconnects when Pi is back.

## Testing

Five layers. Enough to catch regressions, not a cathedral.

1. **Protocol contract** — `schemas/v1/*.json` referenced by both sides. `jsonschema`-based CI validator runs every message in `tests/protocol/fixtures/`.
2. **`pendantd/` unit tests** (pytest) — WebSocket handshake, heartbeat watchdog timing, button config loader (valid/invalid fixtures), Agent SDK tool wrapper (mocked SDK), voice pipeline (canned WAV fixtures, mocked subprocesses).
3. **Firmware unit tests** (`idf.py test`) — LVGL UI render-to-framebuffer with golden hashes, JSON parser fuzz, VAD gate threshold tests.
4. **Hardware-in-loop** (manual, not CI) — dry-run flow pass (`ROBOT_MD_DRY_RUN=1`), documented `docs/proof.md` checklist for a "wet" pre-release test against the real SO-ARM101.
5. **CI** (GitHub Actions) — `pytest` on ubuntu-latest, `idf.py build` for ESP32-C6, schema tests gate both. No hardware in CI. Anthropic API calls recorded with `vcr.py` for a few representative turns.

## Known limits (v0.1)

Documented so they're not surprises:

- **E-stop is software-only.** Heartbeat soft-stop halts the arm on pendant loss; there's no hardware-latched mushroom button in the BOM. The SO-ARM101's wrist_flex can latch into overload (per hardware notes) and no amount of software can clear that — a physical power cycle is needed. v2 adds a hardware e-stop option. Cf. Section 4 row "Servo latched."
- **Single pendant per robot.** The architecture supports multi-pendant (each gets its own Agent SDK session), but sharing one `/dev/ttyACM0` safely across sessions isn't tackled in v0.1. Assume 1:1 for now.
- **No OTA firmware updates.** Flashing is manual (`idf.py flash`). Fine while the pendant is young.
- **No on-pendant wake word in v0.1.** Wake word runs on the Pi via OpenWakeWord; audio-while-speaking leaves the pendant over the LAN even before "Claude" is detected. Same LAN as the robot, no cloud, but worth knowing.
- **Default install ships without a wake-word model.** Ships PTT-only; operator trains + drops in a `claude.tflite` model to opt into wake mode.
- **Custom wake word "Claude"** — OpenWakeWord custom models train locally on the Pi 5 via the project's Python trainer. The training flow is out of scope for this v0.1 spec; a pointer goes in the pendant README once the training recipe is written.
- **Thumbnail is JPEG-at-2fps, not video.** Glanceable, not monitoring-grade.
- **Battery/mounting out of scope.** v0.1 is wall-powered, desk-mounted.
